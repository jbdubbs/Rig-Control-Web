# Dev Notes — RigControl Web

## Current State
A full-stack web application (Express + Vite + Socket.io) designed to control amateur radio equipment via Hamlib's `rigctld`. It features a real-time dashboard with frequency, mode, and meter displays. Recent updates added the ability to automatically start and manage a local `rigctld` process directly from the web interface. The app now supports stable multi-window operation and optimized video streaming.

| Windows Inbound Audio Fix | Root cause of 22–50 second audio gaps was a USB device sample rate mismatch combined with naudiodon's `highWaterMark` blocking: `audioIOAdon.read(N)` waits for exactly N bytes before resolving, so a large `highWaterMark` (16384) caused very long waits. Fix: `framesPerBuffer: 0` (lets PortAudio negotiate native buffer size), `highWaterMark: 256`, `maxQueue: 10`. Device must also be at 44.1 kHz or the same rate that Windows reports; MME/DirectSound resample transparently, WASAPI requires an exact 48 kHz match. | 2026-04-14 |
| Windows Outbound Audio Fix | Every-other-packet audio loss was caused by two concurrent writers to naudiodon's Writable stream: the socket `audio-outbound` handler wrote PCM directly while a `setInterval` silence timer also wrote. On Windows, the naudiodon write mechanism serialises writes and the race between the two sources caused frames to be dropped or mis-ordered. Fix: introduce an outbound jitter buffer (`outboundJitterBuffer[]`); socket handler only decodes Opus and enqueues PCM; the unified 20 ms `setInterval` is the sole writer, draining the buffer when PTT is on and writing silence otherwise. `framesPerBuffer: 0` applied to output stream as well. | 2026-04-14 |

## Known Issues / Tech Debt
- `rigctld` path is assumed to be in the system PATH.
- If `rigctld` is already running outside the app on the same port, the spawned process will fail with an error, which we now catch and display in the log view.
- Split VFO support depends on the specific radio model configured in `rigctld`.
- Some radios (e.g. FT-891) return `RPRT -11` for certain commands in incompatible modes (e.g. NB in FM). These are now handled gracefully as immediate rejections rather than timeouts; no socket destruction occurs.

## Architecture Notes
- **Modular Backend**: `server.ts` is a 269-line orchestrator that wires together 9 focused modules under `server/`: `context.ts` defines the shared `ServerContext` object (passed by reference to every module), `tls.ts` handles cert generation, `settings.ts` manages persistence, `rigctld.ts` spawns/monitors the child process, `rigComm.ts` owns the TCP socket and rig polling, `audio.ts` manages naudiodon I/O, `cw.ts` implements the iambic state machine, `video.ts` handles WebCodecs relay, and `vlog.ts` provides the verbose-logging helper.
- **Modular Frontend**: App.tsx (987 lines, 2 useState calls) is a thin composition root. Business logic lives in 7 custom hooks under `src/hooks/`: `useRigControl`, `useAudio`, `useVideoStream`, `useCWKeyer`, `usePotaSpots`, `useRigctld`, `useCwDecoder`, `useLayoutState`, `usePanelState`. UI structure lives in `src/layouts/` (CompactLayout, DesktopLayout, PhoneLayout, PhoneStickyBar) and `src/modals/` (SettingsModal, VideoSettingsModal). All layout and modal components are wrapped in `React.memo`; all handlers use `useCallback`.
- **Panel Component System**: Each functional UI section (VFO, S-Meter, RF Levels, Controls, etc.) lives in its own file under `src/panels/`. `PanelChrome` (`src/components/PanelChrome.tsx`) provides a consistent collapsible/expandable wrapper with title and optional header-action slots used by all panels. Layout files import and compose panels. This separation of chrome from content is the prerequisite for a future user-configurable drag-and-drop grid layout.
- **Server-as-Manager**: The Express server is not just a proxy but a process manager for `rigctld`.
- **Event-Driven**: All radio state changes flow through Socket.io to ensure low latency.
- **Graceful Shutdown**: The server listens for `SIGINT`/`SIGTERM` to kill the `rigctld` child process, preventing orphaned processes.
- **Log Buffering**: The server keeps the last 100 lines of `rigctld` output in memory to provide context to newly connected clients.
- **Split VFO State**: Managed as a boolean in the rig status, triggering specific UI color overrides.
- **Multi-Window Awareness**: The backend now avoids resetting the rig connection if a new client connects to the same host/port, ensuring stability across multiple tabs.
- **WebCodecs Video Pipeline**: The Electron app is always the video source. It captures via `getUserMedia` + `MediaStreamTrackProcessor`, encodes H.264 (AVCC, avc1.42001F) with `VideoEncoder`, and emits chunks to the server. The server buffers the latest keyframe (with its AVCC description) and relays all chunks to remote clients, which decode with `VideoDecoder` and render to a `<canvas>`. Any client can configure video settings and request start/stop; the server broadcasts changes to the Electron source.
- **Client Audio Routing**: Uses `getUserMedia` with specific `deviceId` for input and `setSinkId` for output, allowing full control over local audio hardware.
- **Native Audio Backend**: Uses `naudiodon` for direct access to host audio devices (ALSA/Pulse/CoreAudio/WASAPI) without spawning external processes, improving reliability and latency.
- **Multi-Client Mic Policy**: Server tracks `activeAudioClientId` based on client interaction events, enforcing a single-source audio stream to the backend.
- **Low-Latency Audio Pipeline**: Uses 48kHz mono Opus encoding via `libopus-node` and WebCodecs, with `AudioWorklet` for capture/playback and a PCM ring buffer for precise frame slicing.
- **POTA Spots Pipeline**: Browser polls `https://api.pota.app/spot/` on a client-side `setInterval`. Raw results are deduplicated by activator callsign (latest `spotTime` wins), filtered by max age (UTC-corrected by appending `'Z'` to bare ISO strings), mode, and band before being passed to a `useMemo`-derived sorted list. `POTA_BANDS` defines 14 amateur bands as kHz min/max ranges. Settings (`enabled`, `pollRate`, `maxAge`, `modeFilter`, `bandFilter`) are persisted to `settings.json` via the existing `save-settings` socket event.
- **SOTA Spots Pipeline**: Fully functional. Browser polls `https://api2.sota.org.uk/api/spots/-1/all` on a client-side `setInterval`. Results are filtered by max age, mode, and band. `SOTA_BANDS` mirrors the same kHz min/max structure as `POTA_BANDS`. Click-to-tune and SSB USB/LSB resolution work identically to POTA. Settings persisted to `settings.json` alongside POTA settings.
- **WWFF Spots Pipeline**: Browser polls the wwff.world API on a client-side `setInterval`. `WwffSpot` type uses `frequency_khz` (number) and `spot_time` (Unix timestamp). Filtered by max age, mode, and band. Click-to-tune resolves SSB to USB/LSB by 10 MHz boundary. Settings persisted under `wwffSettings` in `settings.json`.
- **All Spots Combo Panel**: `SpotComboPanel` (`src/panels/SpotComboPanel.tsx`) is a thin tab shell that accepts render-prop functions for each spot type's table. `ComboSpotSettingsModal` (`src/modals/ComboSpotSettingsModal.tsx`) provides unified per-type settings in a single tabbed modal. Active tab persisted to `localStorage` under `spots-combo-tab`.
- **MUF Map Panel**: `MufMapPanel` (`src/panels/MufMapPanel.tsx`) embeds SVG propagation maps from prop.kc2g.com. Metric (MUFD/foF2) and time-slot (now/−1h/−12h/−24h) are toggled in a controls bar. Zoom uses `width/height = scale * 100%` on the image wrapper for crisp SVG rendering at any zoom level; clamping keeps the image covering the container. Mouse wheel, drag, and pinch-to-zoom are all supported. Panel height is configured at add-time via the PanelPicker and stored in `GridItem.heightPx`.
- **PanelPicker Config Step**: `PanelPicker` uses a two-step flow for panels with configurable options (`PANEL_CONFIG_OPTIONS` in `layout.ts`). Step 1 lists available panels; step 2 shows per-panel options (height slider, full-width toggle). `GridLayoutCallbacks.addPanel` now accepts an optional `PanelAddConfig`; `useLayoutConfig` maps `fullWidth: true` to `w: 9999` so the compact-layout segment renderer treats the panel as a full-width row.
- **General Settings Modal Tabs**: The settings modal uses a tab bar rendered between the header and content area. `activeSettingsTab` state drives which panel is visible. Adding new tabs requires only adding a string literal to the tab array and a corresponding conditional content block.
- **CW Keyer Pipeline**: Client sends `cw-paddle { dit, dah, straight, t }` events (where `t` is ms since socket connect, computed via `performance.now() - cwConnectTimeRef`). Server-side iambic state machine (`iambic-a`, `iambic-b`, `straight`) reconstructs element timing from these offsets, drives a DTR/RTS serial line via `serialport`, and also handles `rigctld-ptt` keying mode. A 5 s stuck-key watchdog fires `emitCwKey(false)` if no key-up event arrives. Client-side sidetone uses a separate Web Audio `AudioContext` (oscillator → gain → destination) for zero-latency local feedback; it is initialized in `initSidetone()` only after `localAudioReady` is true, routed to the selected output device via `setSinkId`, and explicitly resumed. `teardownSidetone()` closes and nulls the context whenever the keyer is disabled or `localAudioReady` goes false, ensuring a clean rebuild on the next enable.
- **Phone CW Paddles**: When `cwSettings.enabled` is true and the rig is in a CW mode (`CW`, `CWR`, `CW-R`), the phone sticky bar swaps the PTT button for side-by-side dit (·) and dah (—) touch buttons. Both use `onPointerDown`/`onPointerUp`/`onPointerCancel` with `setPointerCapture` for reliable touch tracking. They set `ditPressedRef`/`dahPressedRef` and call `emitCwPaddle`, and the existing iambic tick loop drives `sidetoneOn`/`sidetoneOff` in response.
- **App Icons**: Platform-specific icons in `assets/icons/` — `rcw.ico` (Windows), `rcw.icns` (macOS), `rcw_512x512.png` (Linux). Bundled in the asar via the electron-builder `files` array. `BrowserWindow` receives the 512px PNG as its `icon` for Linux runtime taskbar display. `app.setName('RigControl Web')` is called before `whenReady()` to set a clean, predictable `WM_CLASS`.
- **Linux Desktop Integration**: `--install` / `--uninstall` CLI flags allow GNOME integration without AppImageLauncher. The install handler reads the icon from `$APPDIR/resources/app.asar`, writes a `.desktop` file with correct `StartupWMClass`, and refreshes icon/desktop caches. Both handlers call `process.exit()` before `app.whenReady()` — no window or GUI is created.
- **Local Speaker Volume**: A `GainNode` (`inboundGainRef`) sits between the `playback-processor` AudioWorkletNode and `AudioContext.destination`. The UI slider (0–200%) drives `gain.value` in real time. Above 100% adds amplification beyond the system volume level; below 100% attenuates. Value persisted in `localStorage` under `local-audio-inbound-volume`.
- **CW Decoder Pipeline**: `GGMorseDecoder` (wrapping GGMorse WASM) runs entirely in the browser. After WebCodecs decodes each Opus packet to F32 PCM, the raw float array is also passed to `cwDecoderRef.current.processSamples()`. This happens unconditionally when `cwDecodeEnabledRef` is true — the mute flag only gates speaker playback, not the decoder feed. WASM is loaded lazily (once) on first toggle; `reset()` is called on disable to flush internal state without unloading the module. Decoded characters append to `cwDecodedText` (capped at 2000 chars), auto-scrolled to the bottom. `cwStats` carries pitch (Hz) and speed (WPM) from the last decoded burst, displayed in a small inline readout next to the text area. The decoder display appears inline in compact/desktop layouts and as a collapsible section in the phone layout.
